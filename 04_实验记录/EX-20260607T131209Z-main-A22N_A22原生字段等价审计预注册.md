## 新手解释

前面我们找到的最好候选是 A22 cap70：高分不再完全过滤，而是把风险仓位压到 70%。但旧 A22 配置其实借用了早期 A13 的字段名，所以日志会写成 `A13_SCORE56_BUDGET`。这会让后续上线、审计和排错变得不清楚。

本实验不寻找更高收益，也不调新参数。它只问一个工程问题：把同一个 A22 cap70 逻辑从旧 A13 字段迁移到原生 A22 字段后，回测结果是否完全一样。

如果完全一样，说明我们可以清理字段和日志语义；如果不一样，说明原生实现和旧实现存在行为差异，不能继续推进默认切换。

## 研究问题

在同一个实验、同一代码状态、同一数据环境中，`legacy_a13_cap70` 与 `native_a22_cap70` 是否能产生完全一致的交易路径和组合结果？

## 假设

### 正向假设

`native_a22_cap70` 只是把旧 A22 cap70 的字段表达从 `r010a13_score56_*` 映射为 `r010a22_hot_score_budget_*`，不改变信号、交易、仓位或成本。因此它应与 `legacy_a13_cap70` 在四个历史分段上完全等价。

### 反向假设

如果原生 A22 字段在调用顺序、触发条件、目标 ETF 检查、风险仓计算或日志副作用上与旧 A13 字段不同，则两组结果会出现 final、MDD、交易数、费用或关键文件 hash 差异。

## 样本划分

| 分段 | 用途 |
| --- | --- |
| `2020_2021` | 压力与高分过热早期样本 |
| `2022_2023` | 回撤和震荡样本 |
| `2024` | 负控段，历史上 A22 与 hard5 多次相同 |
| `2025_20260519` | 近端强趋势样本 |

## 实验设计

| variant | 含义 | 配置来源 |
| --- | --- | --- |
| `legacy_a13_cap70` | 旧字段表达，保留 `r010a13_score56_risk_budget_enabled=true`、`risk_cap=0.70` | `configs/research/R010-A22/hot_score_budget/a20_score_ge5_cap70/` |
| `native_a22_cap70` | 原生字段表达，关闭 A13，开启 `r010a22_hot_score_budget_enabled=true`、`base_risk_cap=0.70`、`state_tier_enabled=false` | 由同一旧配置转换生成 |

固定条件：

- 不改变 `max_score=9999` 和 `score_hot_filter_mode=conditional_hot_state`。
- 不改变交易成本、回测区间、数据需求、B3/A2 等其他策略字段。
- 不启用 A23 的 broad-blowoff 分层。
- 同一实验中同时跑旧字段版和原生字段版，避免跨时间环境漂移。

## 通过门槛

必须全部满足：

- 8/8 回测完成并有 `summary.json`。
- 8/8 有 run log、manifest，且 `exit_status=0`。
- 8/8 配置字段审计通过。
- 原生 vs 旧字段 4/4 分段 `final_value` 完全一致。
- 原生 vs 旧字段 4/4 分段 `max_drawdown` 完全一致。
- 原生 vs 旧字段 4/4 分段交易数和费用完全一致。
- `equity_curve.csv`、`positions.csv`、`subportfolio_equity.csv` hash 4/4 完全一致。
- 忽略 `order_id` 后的 `trades.csv` 归一化 hash 4/4 完全一致。

## 失败条件

出现任一情况即失败：

- 任一分段缺少结果、日志、manifest 或 exit status 非 0。
- 配置字段映射不满足旧 A13 与原生 A22 的一一对应。
- 任一分段 final、MDD、交易数、费用或关键 hash 不一致。
- 日志出现 `Traceback`、`ERROR`、`SystemError` 等风险标记。

失败后不允许把原生 A22 作为 hard5 替代候选继续推进；必须先定位代码差异。
