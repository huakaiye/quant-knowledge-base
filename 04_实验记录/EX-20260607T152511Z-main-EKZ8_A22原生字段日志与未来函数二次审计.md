## 1. 新手摘要

这次实验想知道：A22N 证明结果文件完全一致后，日志语义和未来函数边界是否也干净。

我们原本预计：native A22 版本应只出现 `A22_HOT_BUDGET` 指纹，不再出现 `A13_SCORE56_BUDGET`；每个动作日志里的 `signal_date` 应不晚于 `trade_date`；legacy 与 native 的预算触发次数应该按分段一一相同。

实际看到：待只读脚本审计。

这说明：待补。

但还不能说明：即使通过，也只说明 A22N 这组历史回测日志和日期边界没有发现问题；它不等于实盘或 shadow 已可上线。

下一步要做：运行只读审计脚本，汇总配置字段、manifest、run log、`logs.jsonl` 指纹和日期顺序。

## 2. 研究背景

A22N 已经证明 legacy A13 字段表达与 native A22 flat cap70 在四段结果文件上完全等价。剩余问题是：字段迁移后，日志是否真的从 A13 语义切到 A22 语义；A22 预算触发动作是否存在 `signal_date > trade_date` 的未来函数迹象；以及 manifest/config sha 是否能支持后续工程清理。

本实验只读 A22N 的既有产物，不运行新回测、不更新数据、不改变策略逻辑。

## 3. 实验前假设

`native_a22_cap70` 在 A22N 结果中应表现为日志语义上的原生 A22：配置关闭 A13、开启 A22；触发事件使用 `A22_HOT_BUDGET`；日期顺序不出现未来函数；触发次数与 legacy A13 字段表达按分段完全相同。

## 4. 实验前预测

如果假设为真，应该看到：

- 指标：8/8 配置字段通过；8/8 manifest 存在 `config_sha256`；8/8 run log `exit_status=0` 且无风险关键词。
- 交易行为：native 四段 `A22_HOT_BUDGET` 触发次数分别应等于 legacy A13 的对应触发次数；native 不应出现 `A13_SCORE56_BUDGET`。
- 风险表现：本实验不新增收益判断，只复核 A22N strict 已通过的结果路径。
- 分段表现：2020_2021、2022_2023、2025_20260519 应看到预算触发，2024 可能仍为 0 触发负控。

## 6. 竞争性解释

即使结果符合预期，也可能是：

- `logs.jsonl` 中不同模块的日志格式不完全一致，字符串指纹只能作为语义证据，不能替代代码审计。
- 2024 无触发只能证明无触发负控，不提供触发路径证据。
- `signal_date <= trade_date` 只能排除日志中可见的未来日期，不能证明所有数据源都已完成实盘级可得日审计。
- manifest 记录 config sha，但若不和台账锁定版本，仍可能出现后续配置漂移。

## 7. 证伪条件

出现以下情况，本假设不通过：

- native 配置仍启用 `r010a13_score56_risk_budget_enabled=true`。
- native `logs.jsonl` 出现 `A13_SCORE56_BUDGET` 或 `A13 score5-6风险预算触发`。
- native 四段 `A22_HOT_BUDGET` 触发次数与 legacy A13 对应分段不一致。
- 任一 R010-A/R010-B/R010-B1 日志存在 `signal_date > trade_date`。
- 任一 run log 缺失、`exit_status` 非 0 或出现 `Traceback/ERROR/SystemError`。
