# Obsidian 双向链接规范

Obsidian 的双向链接应当充分使用，但不能替代 ID、frontmatter 和台账。

## 基本原则

| 机制 | 职责 |
| --- | --- |
| `[[双向链接]]` | 给人类阅读、跳转、反向链接和局部图谱使用 |
| frontmatter ID | 给 Agent、脚本和 Git 合并校验使用 |
| `01_台账/` | 给全局检索、状态过滤和资产盘点使用 |
| Canvas | 给全局路线视觉导航使用 |

一句话：双向链接负责“读起来顺”，ID 和台账负责“管得住”。

## 链接格式

优先使用带路径和别名的链接：

```markdown
[[02_研究方向/RD-20260605T115651Z-main-DP00_双池轮动策略|双池轮动策略]]
```

原因：

- 路径明确，避免同名文件冲突。
- 别名短，阅读体验更好。
- Obsidian 重命名文件时通常能自动更新链接。

## 表格中的限制

不要在 Markdown 表格中使用 `[[目标|别名]]`。表格会把 `|` 当作列分隔符，导致双向链接或表格结构异常。

表格中如必须链接文件，使用不带别名的双链：

```markdown
| 方向 | 状态 |
| --- | --- |
| [[02_研究方向/RD-20260605T115651Z-main-DP00_双池轮动策略]] | active |
```

如果需要可读的短别名，优先改成正文列表，而不是表格。

## 必须互链的资产

| 当前文档 | 必须链接到 |
| --- | --- |
| 研究方向 | 父方向、子方向、策略档案、当前最佳实验、当前决策 |
| 实验记录 | 研究方向、策略档案、使用的文献或灵感、产生的决策 |
| 研究决策 | 受影响方向、关键实验、上一张决策、后续动作 |
| 文献卡 | 因子卡、数据卡、机制卡、相关研究方向 |
| 因子/数据/机制/模块灵感 | 来源文献、相关方向、升级后的实验 |
| 策略档案 | 策略级方向、模块方向、平台协作规范 |
| 术语词条 | 相关方向、实验或方法论 |
| 迁移卡 | 旧库来源说明、新库目标资产 |

## 链接密度

不要把每个词都做成链接。推荐：

- 每个关键资产首次出现时链接一次。
- 同一段落内不要重复链接同一个目标。
- 表格中的 ID 可以用 `[[路径|ID]]`，正文解释用 `[[路径|中文名]]`。
- 高频术语只在首次出现或容易误解时链接到术语库。

## 反向链接怎么用

新 Agent 接手某个方向时，先打开方向页，然后查看 Obsidian 的 Backlinks 面板：

1. 找到所有引用该方向的实验记录。
2. 找到所有引用该方向的决策卡。
3. 找到关联文献、因子或模块灵感。
4. 再对照 `01_台账/` 检查是否有遗漏。

反向链接是发现上下文的工具，不是最终事实源。若反向链接和台账冲突，必须回到正文和台账一起修正。

## Canvas 和双链的关系

Canvas 节点应链接到真实文件，不创建没有正文的孤立节点。Canvas 表示“路线结构”，双向链接表示“文档关系”。

当一个方向的反向链接很多时：

- 全局 Canvas 只保留方向和关键决策。
- 普通实验继续通过反向链接和实验台账检索。
- 必要时为该方向新建子 Canvas。

## GitHub 兼容

GitHub 不原生解析 Obsidian 双链，所以关键入口仍应保留普通 Markdown 路径或台账路径。不要只依赖双链。

## Agent 写作要求

Agent 新建或修改核心资产时，必须检查：

- frontmatter ID 是否存在。
- 正文是否有“关联链接”区块。
- 方向、实验、决策之间是否形成双向可追踪关系。
- 台账是否同步更新。

缺少双向链接不一定影响脚本运行，但会显著降低人类接手效率。
