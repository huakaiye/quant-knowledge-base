# 研究方法论

本库的研究目标不是把回测曲线调好看，而是持续积累可解释、可复现、可证伪的策略知识。

## 核心原则

1. 先有假设，再跑实验。
2. 先定义失败条件，再看结果。
3. 先比较基准，再判断改进。
4. 先解释反证，再升级结论。
5. 先记录研究理由，再修改方向。
6. 先通过未来函数和过拟合审计，再升级结论。

## 标准研究链路

```text
文献或观察
-> 机制灵感
-> 研究方向
-> 子代理计划或豁免
-> 实验前假设
-> 子查核对路径、字段和已有证据
-> 回测或诊断
-> 子研归纳支持证据、反对证据和边界
-> 实验记录
-> 决策卡
-> 主控复核未来函数、过拟合和路线影响
-> 方向页和驾驶舱更新
```

每个环节都要能被后来者追溯。

子代理不是替代主控判断，而是把检索、核对、归纳和受控执行变成可审计分工。正式实验或影响路线的决策在进入 `completed` 前，必须具备子代理调用记录，或写明子代理豁免原因。

## 实验前必须写清

| 项目 | 新手解释 |
| --- | --- |
| 研究信念 | 我们现在相信市场或策略中存在什么规律 |
| 本次假设 | 这次只验证哪一个具体想法 |
| 实验前预测 | 如果假设是真的，指标或行为应该怎样变化 |
| 基准对照 | 和谁比才算有意义 |
| 竞争性解释 | 即使结果变好，也可能是哪些别的原因导致 |
| 证伪条件 | 出现什么结果就说明假设不成立或暂不值得继续 |
| 样本划分 | 哪些区间是样本内、样本外、压力期 |
| 子代理计划 | 哪些检索、核对或归纳任务交给子代理；若不调用，为什么可以豁免 |

## 实验后必须写清

| 项目 | 新手解释 |
| --- | --- |
| 实际观察 | 指标和行为实际发生了什么 |
| 支持证据 | 哪些结果支持原假设 |
| 反对证据 | 哪些结果反驳原假设或暴露风险 |
| 偏差诊断 | 预测和实际不一致时，可能原因是什么 |
| 可信度变化 | 这次实验让我们更相信还是更怀疑 |
| 下一步 | 下一轮最该减少哪一个不确定性 |
| 新手总结 | 用普通语言解释这轮研究为什么这样判断 |
| 子代理记录 | 子代理实际做了什么、主控如何复核、哪些结论不能外推 |

## 决策状态

| 状态 | 含义 | 典型后续 |
| --- | --- | --- |
| `promote` | 证据足够强，可以进入更高层级验证或候选合并 | 样本外、影子观察、组合级验证 |
| `promote_candidate` | 初步有效，但还缺关键审计 | 补成本、消融、未来函数或压力期 |
| `revise` | 方向仍有价值，但当前实现需要修改 | 收窄假设，重写实验 |
| `park` | 暂停，不是失败，只是当前优先级不足或证据不足 | 保留证据，等待新数据或新机制 |
| `kill` | 假设被关键证据否定，短期停止投入 | 归档并写清失败原因 |
| `observe` | 只观察，不改变默认策略或实盘边界 | 记录监控指标 |

## 通过标准

一个策略改进不能只因为收益更高就通过。至少要检查：

- 是否优于明确基准。
- 是否在样本外仍有解释力。
- 是否经过交易成本和滑点扰动。
- 是否做过消融，证明收益来自目标模块。
- 是否没有使用未来数据。
- 是否在关键压力期没有引入不可接受风险。
- 是否降低了组合层面的真实不确定性。

详细审计标准见 `08_方法论/研究质量审计规范.md`。缺少未来函数审计或过拟合审计的实验，只能作为观察材料，不能作为路线升级依据。

## 对新手友好的总结写法

每份实验记录最后必须有一段“给新手的结论”，建议格式：

```text
这次实验想知道：……
我们原本预计：……
实际看到：……
这说明：……
但还不能说明：……
下一步要做：……
```

这段不追求学术化，追求让完全没读过前文的人也能理解研究方向为什么继续、修改、暂停或淘汰。
