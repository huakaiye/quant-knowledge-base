~~~markdown
## 命令与编码硬规则

所有 Agent 和子代理都必须默认按 UTF-8 处理中文文档、JSON 和 Markdown。PowerShell 中读取或输出中文前，优先显式设置输出编码：

```powershell
[Console]::OutputEncoding=[System.Text.UTF8Encoding]::new($false)
```

PowerShell 引号规则必须特别注意：

- 路径中包含中文、空格、全角符号或方括号时，优先用单引号包裹路径。
- `Get-Content`、`Test-Path`、`Remove-Item`、`Set-Location` 等支持时优先使用 `-LiteralPath`，避免 `[]` 被当成通配符。
- 读取 Markdown 时显式使用 `-Encoding UTF8`。
- 避免把复杂命令嵌入多层双引号；涉及 WSL、Python、JSON 或正则时，优先把参数拆开，或使用配置文件、脚本文件、数组参数。
- 不要在 PowerShell 中拼接长字符串后交给 `cmd /c` 执行；能用 PowerShell 原生命令就用原生命令。
- 检索优先使用 `rg`；读取少量文件用 `Get-Content -Encoding UTF8`；复杂结构化处理再使用脚本。

### 回测进度可见性硬规则

子代理运行回测时，必须让主线程和用户可以看到实时进度。不得只使用静默重定向：

```bash
PYTHONPATH=src python3 src/run_v2_backtest.py --config xxx.json > xxx.run.log 2>&1
```

推荐使用 `tee`，做到终端实时显示，同时落盘日志：

```bash
PYTHONUNBUFFERED=1 PYTHONPATH=src python3 src/run_v2_backtest.py --config xxx.json 2>&1 | tee results/v2/research/<research_id>/<run_name>.run.log
```

若需要后台运行，也必须同时提供可追踪的进度命令和日志路径，例如：

```bash
tail -f results/v2/research/<research_id>/<run_name>.run.log
```

子代理最终汇报必须列出：

- 当前正在运行的回测配置。
- WSL PID。
- 实时日志路径。
- 可复制的 `tail -f` 进度查看命令。
- 已完成/缺失配置数。

批量脚本也应遵守该规则：每个配置输出独立 `.run.log`，并使用 `tee` 保持进度可见。

典型安全读文件写法：

```powershell
[Console]::OutputEncoding=[System.Text.UTF8Encoding]::new($false)
$p='E:\【笔记库】\量化研究库\📊 驾驶舱.md'
Get-Content -LiteralPath $p -Encoding UTF8
```

## `gpt-5.3-codex-spark` 并行策略

用户当前用量口径中，`gpt-5.3-codex-spark` 可按无限用量处理。因此，主 Agent 可以主动把互不依赖的简单重复劳动拆成多个 `gpt-5.3-codex-spark` 子代理并行执行。

允许并行的典型场景：

- 一个子代理检索研究记录，另一个检索决策卡片，第三个检索概念词条。
- 一个子代理检查 frontmatter，另一个检查双链，第三个检查表格。
- 多个子代理分别阅读不同专题页，只摘录事实、路径和关键句。
- 多个子代理分别汇总不同结果目录或配置目录的文件清单。

并行边界：

- 子任务必须相互独立，输出可合并。
- 不要让多个 5.3 子代理重复分析同一个未拆分问题。
- 5.3 子代理默认只读；如确需写文件，必须明确限定互不重叠的写入范围，并由主 Agent 复核。
- 5.3 子代理不得做路线判断、实盘判断、未来函数最终审计或正式实验结论。
- 若系统实际模型配额或可用性发生变化，以当前工具返回的可用模型为准。

## 子代理命名规则

主 Agent 调用子代理时，必须在委派提示词第一行写明任务代号。即使工具没有显式命名参数，也要在 prompt 中声明，要求子代理最终回复时复述该任务代号。

命名格式：

```text
<中文层级前缀>-<三位序号>-<中文任务短名>
```

统一前缀：

| 前缀 | 含义 | 示例 |
| --- | --- | --- |
| `子查` | `gpt-5.3-codex-spark` 简单检索、清单、格式检查、路径核对 | `子查-001-研究记录检索` |
| `子研` | `gpt-5.4` 中等复杂阅读、证据归纳、草稿整理 | `子研-001-证据链归纳` |
| `子执` | 需要执行脚本或回测且必须展示进度的受控执行任务 | `子执-001-可见进度回测` |
| `主控` | 主 Agent 自己保留的关键路径和最终判断任务 | `主控-001-路线终审` |

命名要求：

- 任务短名优先使用中文，要求短、具体、可读。
- 中文任务短名只用于 Agent prompt、汇报和研究库文档；如果必须写入文件名、日志名或 shell 变量，可另附 ASCII `run_id`。
- 同一轮并行任务序号不可重复。
- 子代理最终回复必须列出 `任务代号`、`使用模型`、`读取文件`、`修改文件`、`执行命令` 和 `结论边界`。
- 简单并行任务尽量按对象拆分，例如 `子查-001-实验记录检索`、`子查-002-决策卡检索`、`子查-003-链接检查`。
- 如果子代理被要求写文件，任务名必须体现写入范围，例如 `子研-002-R010B5R4草稿整理`，且不得与其他子代理写入范围重叠。
- 旧格式 `S53/M54/CTL` 只作为历史别名保留，不再作为新任务首选命名。

## 调度原则

每轮任务开始前，主 Agent 先做轻量拆解：

1. 哪些任务在关键路径上，必须主 Agent 自己做。
2. 哪些任务是简单重复劳动，可以拆成多个 `gpt-5.3-codex-spark` 子代理并行处理。
3. 哪些任务是中等复杂阅读、归纳、草拟，可以交给 `gpt-5.4`。
4. 哪些任务涉及路线、实盘、未来函数或正式实验结论，必须主 Agent 最终把控。

只有当子代理任务能并行推进、边界清晰、输出可复核时才调用子代理。不要为了形式调用子代理。

## 可交给 `gpt-5.4` 的任务

适合下放的中等复杂任务：

- 阅读一组研究记录并提炼证据链。
- 比较两个研究节点相比上一个节点改变了什么。
- 草拟研究记录、研究决策卡片、路线快照更新段落。
- 梳理支持证据、反对证据、竞争性解释和证伪测试。
- 初步建议 `observe / revise / promote / park` 措辞，但不能作为最终判断。

委派示例：

```text

## 主 Agent 必须亲自处理的任务

以下任务不得完全交给子代理：

- 是否改变研究主线。
- 是否产生 `kill / revise / promote / park / observe / promote_candidate` 等方向判断。
- 是否进入实盘、影子观察、真实路径回测或参数敏感性测试。
- 是否修改 MiniQMT、平台配置或实盘默认开关。
- 是否认为某个结果通过未来函数审计。
- 是否建立新的研究信念或正式研究决策。
- 最终写入研究库前的事实核验、反证核验和边界确认。

## 子代理输出复核清单

主 Agent 使用子代理结果前至少复核：

- 文件路径是否真实存在。
- 引用是否来自当前研究库，而不是旧对话记忆。
- 是否混淆代理审计、正式回测、真实路径验证和实盘结果。
- 是否违反不自动回测、不自动更新数据、不自动改实盘的边界。
- 是否把只读标签误升级为交易信号。
- 是否用后验阈值拼接路线。
- 是否遗漏反证、样本外、成本、消融和未来函数审计。
- 是否需要同步更新 [[研究库健康检查]]。

## 写入规则

子代理可以草拟，但主 Agent 负责最终写入。

若只是补充接手说明、清单或非路线性文档，可更新入口页和 [[研究库健康检查]]，通常不需要新建研究决策卡片。

若发生方向决策，主 Agent 必须确保：

1. 新增或更新 `🔀 决策/` 下的研究决策卡片。
2. 同步更新对应 `🧭 方向/` 页面。
3. 若影响主链、分支或当前候选，同步更新 `📊 研究全局路线.canvas`。
4. 必要时更新 [[📊 驾驶舱]] 和 [[研究资产清单]]。
5. 新增或修改 Obsidian 链接结构后，运行全库格式检查和双链检查，并更新 [[研究库健康检查]]。

## 推荐工作流

1. 主 Agent 读取入口文档，确认当前主线和 `next_action`。
2. 主 Agent 拆解任务，区分关键路径和可并行任务。
3. 简单检索和清单任务优先拆给多个 `gpt-5.3-codex-spark` 子代理并行处理。
4. 中等复杂归纳或草拟任务交给 `gpt-5.4`。
5. 主 Agent 并行处理关键路径。
6. 汇总子代理结果。
7. 主 Agent 复核事实、证据、反证和边界。
8. 若需要写入，按研究库规范更新文档。
9. 完成后说明修改了哪些文件、当前结论和下一步。
~~~
