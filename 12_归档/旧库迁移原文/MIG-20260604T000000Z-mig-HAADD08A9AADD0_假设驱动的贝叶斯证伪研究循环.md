~~~markdown
# 假设驱动的贝叶斯证伪研究循环

本方法论用于约束 Agent 独立开展量化策略研究。核心目标不是让 Agent 直接优化回测收益，而是让 Agent 通过实验持续减少对市场机制、信号表达、交易实现和风险来源的不确定性。

一句话原则：

> Agent 每一轮研究都必须回答：我原本相信什么，实验让我看到了什么，哪些观察支持或反驳原假设，下一轮哪个实验最能减少不确定性。

## 核心循环

```text
观察现象
  -> 溯因提出多个机制解释
  -> 写成可检验假设
  -> 推导可观察预测
  -> 设计证伪和区分性实验
  -> 执行回测与诊断
  -> 比较预期与实际
  -> 解释偏差和竞争性解释
  -> 更新研究信念
  -> kill / revise / promote / park
  -> 写入研究记忆
```

## 基本对象

| 对象 | 含义 | 存放方式 |
| --- | --- | --- |
| 研究信念 | 对某个市场机制或策略机制的当前理解 | 写入方向页或实验记录的“当前信念/核心假设”章节 |
| 假设 | 一条可以被实验支持或反驳的具体断言 | 写入研究记录 |
| 预测 | 如果假设成立，应该观察到的结果 | 写入研究记录的实验前部分 |
| 实验 | 为区分解释、验证预测或证伪假设而设计的回测/诊断 | 配置在平台，结论写入研究库 |
| 反思 | 实验结果与预期的差异解释 | 写入研究记录的实验后部分 |
| 决策 | kill / revise / promote / park | 写入 YAML `decision`、正文结论；若改变方向或边界，另建 [[决策链路说明]] 所要求的决策卡片 |

## 实验前必须完成

每个正式实验必须在回测前写清楚以下内容：

| 项目 | 要求 |
| --- | --- |
| 研究信念 | 当前研究方向到底相信什么机制 |
| 先验置信度 | 用低/中/高或 0-1 表示，但不要假装精确 |
| 具体假设 | 本次实验要验证的单一主张 |
| 可观察预测 | 如果假设成立，应当看到什么 |
| 竞争性解释 | 还有哪些更简单或更危险的解释 |
| 证伪测试 | 哪些结果会推翻或显著削弱假设 |
| 决策规则 | 什么条件下 kill、revise、promote |

不满足这些条件的回测只能算探索，不得标记为 `已采纳`。

## 实验后必须完成

回测完成后，Agent 不得直接开始调参。必须先完成反思：

| 维度 | 问题 |
| --- | --- |
| 预测一致性 | 哪些结果符合预期，哪些结果与预期冲突 |
| 最大意外 | 最出乎意料的观察是什么 |
| 偏差来源 | 偏差来自机制、信号、样本、成本、组合暴露还是数据口径 |
| 竞争性解释 | 有没有比原假设更简单的解释 |
| 证伪结果 | 哪些证伪测试通过，哪些失败 |
| 信念更新 | 置信度上升、下降还是保持 |
| 下一轮实验 | 下一轮实验是为了提高收益，还是为了减少不确定性 |

## 偏差诊断维度

Agent 在解释实验偏差时，必须逐项检查：

| 维度 | 检查问题 |
| --- | --- |
| 数据偏差 | 是否有前视、幸存者偏差、公告时点错误、复权口径错误 |
| 机制偏差 | 原始机制是否不完整，信号是否只是弱代理变量 |
| 样本偏差 | 是否只在某些年份、行业、品种、市值段或市场状态有效 |
| 交易偏差 | 成本、滑点、成交约束、停牌、涨跌停是否吞噬收益 |
| 组合偏差 | 是否来自 beta、行业、size、value、momentum 等暴露 |
| 统计偏差 | 是否多重检验、参数不稳定、少数极端日贡献过高 |
| 边际贡献 | 是否与已有因子或策略高度重复 |

## 决策状态

| 状态 | 使用条件 | 下一步 |
| --- | --- | --- |
| kill | 关键预测被反驳，或交易实现后收益消失，且无合理机制修正 | 停止研究，保留失败记录 |
| revise | 机制可能存在，但信号表达、样本定义或执行方式有问题 | 改写假设或信号，设计区分性实验 |
| promote | 样本外、消融、成本和稳健性结果支持假设 | 进入更严格验证或仿真候选 |
| park | 平台、数据或样本暂不支持判断 | 暂停，写清楚恢复条件 |

## 方向决策卡片

只要一次判断会改变研究方向、候选优先级、交易边界、实盘观察边界或后续实验顺序，就必须新建或更新 `🔀 决策/` 下的研究决策卡片。

决策卡片不是实验记录的替代品，而是路线图节点。它必须写清：

| 字段 | 要求 |
| --- | --- |
| 是什么 | 用非参与者也能理解的语言解释该节点，例如某个 cap、gate、filter 或 marker 的含义 |
| 相比上个节点 | 说明新增、删除或改变了哪一层逻辑，不只写参数名 |
| 当前判断 | 明确 `kill / revise / promote / park / observe` 及边界 |
| 证据与反证 | 同时列出支持和反对证据，避免只保留成功叙事 |
| 下一步 | 说明下一轮最能减少哪类不确定性 |

写入决策卡片后，必须同步更新 [[📊 驾驶舱]]；如果主链、分支或当前候选发生变化，还必须更新 `📊 研究全局路线.canvas`。

## 允许的改进行为

Agent 每一轮只能选择少数改进操作，并说明原因：

| 类别 | 合法操作 |
| --- | --- |
| 修正假设 | 改写机制边界、适用市场状态、失效条件 |
| 修正信号 | 换代理变量、去极值、标准化、中性化、滞后处理 |
| 修正实验 | 改样本切分、补消融、补样本外、补成本口径 |
| 修正组合 | 控制权重、换手、风险暴露、容量和再平衡频率 |
| 终止方向 | 记录失败原因，不继续消耗实验预算 |

禁止把所有问题都处理成参数扫描。参数扫描只能回答局部敏感性，不能替代机制解释。

## 平台协作规则

- 研究库只保存假设、设计、路径引用、关键结果和解释。
- 回测配置保存到 `E:\量化平台_V1.4.0\configs\research\<research_id>\`。
- 回测结果保存到 `E:\量化平台_V1.4.0\results\v2\research\<research_id>\`。
- CLI 回测必须按平台规则在 WSL 内运行，先读 `E:\量化平台_V1.4.0\docs\AGENT_RULES.md`。
- 不自动执行数据更新；只有用户明确要求更新数据时才运行更新入口。

## Agent 硬规则

1. 每次实验必须有实验前预测。
2. 每次实验必须有实验后偏差诊断。
3. 结果不好时，先解释偏差，再决定是否改进。
4. 结果很好时，先寻找反证，再决定是否升级。
5. 不只保存成功实验；失败实验是研究记忆的一部分。
6. 下一轮实验优先选择最能区分竞争性解释的设计，而不是最可能提高 Sharpe 的设计。
7. 没有样本外、成本和消融验证的研究最多标记为 `观察中`。
8. 每次方向决策必须有研究决策卡片；改变主链、分支或当前候选时必须同步 [[📊 驾驶舱]] 和 `📊 研究全局路线.canvas`。没有卡片的路线判断视为未完成交接。
~~~
