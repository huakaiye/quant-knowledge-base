# 研究库重构审计与二期方案

本文用于回答一个问题：当前研究库怎样重构，才能让完全不了解项目的新手快速知道研究进度，并且让几年后的研究者仍能找到今天的实验记录、配置、数据、结果分析和决策链。

## 三、目标结构

### 3.1 唯一事实源

重构后应有明确职责：

1. 驾驶舱是唯一新手入口，回答现在是什么状态。
2. 方向页是活文档，回答这个方向为什么走到当前结论。
3. 实验记录是证据文档，回答这次实验怎么设计、怎么执行、观察到什么。
4. 决策卡是路线节点，回答这次证据如何改变研究方向。
5. 资产清单是检索层，回答配置、结果、manifest、summary、CSV、脚本在哪里。
6. 归档只保存历史上下文，不作为当前入口。

### 3.2 新手 10 分钟接手路径

新手不应先读大量旧实验。默认路径应固定为：

1. 读术语词典，解决缩写。
2. 读驾驶舱，知道当前 live、revise、park、waiting、kill。
3. 从驾驶舱进入一个方向页。
4. 在方向页只看三块：当前判断、下一步、关键证据链。
5. 点击最近一张决策卡，理解为什么现在是这个判断。
6. 点击关联实验记录，查看关键指标和资产路径。
7. 如需复现，再从资产清单进入平台配置和结果目录。

## 四、编号、命名和中文优先规范

本节是二期重构必须补齐的硬规范。目标是同时满足两个要求：

1. 机器可检索、可排序、可生成资产清单。
2. 新手可阅读，适合中文表达的地方优先使用中文。

### 4.1 总原则

编号用于机器和长期追踪，标题用于人读。

必须稳定的编号字段使用 ASCII：

```text
research_id
decision_id
direction_id
topic_id
run_id
manifest_id
asset_id
```

文件名和正文标题优先中文：

```text
R20260603-057_R010B5S4_滚动波动率中位参考分组前向样本可用性审计.md
D20260603-R010B5S4-前向样本可用性审计完成.md
非线性仓位控制.md
ETF双池防守系统.md
```

禁止把可读标题写成全英文缩写堆叠。英文缩写可以保留在编号和术语中，但第一次出现必须给出中文解释。

### 4.2 研究方向编号

研究方向编号使用 direction_id。它表示一条可持续推进的研究主线，不等于单次实验。

现有编号继续沿用，不强制重命名历史：

```text
R010-B4：B3 gate 强度分层研究
R010-B5：非线性仓位控制
M007：小盘专用状态标签构造与归因
R026：防守池吸血信号
```

新方向编号建议：

```text
R<三位主线号>-<分支号>
M<三位模块号>
E<三位事件研究号>
G<三位治理号>
```

示例：

```text
R011-A：ETF防守新主线
M008：多策略状态标签新模块
E001：涨停事件价量结构
G001：研究库资产治理
```

编号只负责稳定定位，方向页标题必须使用中文。

### 4.3 主题节点编号

topic_id 表示方向内的具体节点、子实验组或阶段。

规则：

```text
<direction_id><阶段号>
<direction_id><阶段号>R<修订号>
<direction_id><阶段号>S<组件审计号>
```

示例：

```text
R010-B5R1：非线性组合回撤预算代理审计
R010-B5R2：真实路径 2024 Smoke
R010-B5S4：前向样本可用性审计
M007-R1：小盘标签定义修订
```

topic_id 不能替代 research_id。一个 topic_id 可以对应多篇实验记录、多张决策卡和多个平台结果目录。

### 4.4 实验记录编号

research_id 是实验记录的永久编号。

规则：

```text
RYYYYMMDD-NNN
```

示例：

```text
R20260603-057
R20260604-058
```

文件名规则：

```text
<research_id>_<topic_id>_<中文短标题>.md
```

示例：

```text
R20260603-057_R010B5S4_滚动波动率中位参考分组前向样本可用性审计.md
```

规则说明：

1. research_id 只按日期和序号生成，不夹带方向含义。
2. topic_id 单独写入 frontmatter 和文件名。
3. 同一天序号必须唯一。
4. 中文短标题描述实验目的，不写空泛词。
5. 预注册、只读审计、回测、失败运行都可以是实验记录，但必须有 result_status。

### 4.5 决策卡编号

decision_id 是决策卡永久编号。

建议字段值：

```text
DYYYYMMDD-<topic_id>-<ascii_slug>
```

文件名优先中文：

```text
DYYYYMMDD-<topic_id>-<中文短标题>.md
```

示例：

```text
decision_id: D20260603-R010B5S4-forward-sample-readiness
文件名：D20260603-R010B5S4-前向样本可用性审计完成.md
```

规则说明：

1. decision_id 用 ASCII，方便脚本和资产清单。
2. 文件名用中文，方便新手浏览。
3. 一次方向判断必须有决策卡。
4. 一次小修小补不改变路线时，不强制新建决策卡。
5. 决策卡必须写清允许什么、禁止什么、下一步是什么。

### 4.6 平台运行和结果编号

run_id 使用平台现有 manifest 中的 run 标识，不在研究库手工发明。

新增平台结果目录建议：

```text
results\v2\research\<direction_id>\<research_id>_<topic_id>_<英文短主题>
configs\research\<direction_id>\<research_id>_<topic_id>
```

示例：

```text
results\v2\research\R010-B5\R20260603-057_R010B5S4_forward_sample_readiness
configs\research\R010-B5\R20260603-057_R010B5S4
```

说明：

1. 平台路径可以使用英文短主题，减少脚本、WSL 和日志路径兼容风险。
2. 研究库文件名和正文标题仍优先中文。
3. 存量平台目录不强制搬迁，先通过资产清单登记。
4. 如果没有 manifest 或 summary，资产清单必须标记缺口。

### 4.7 文献、想法、概念和策略命名

文献笔记：

```text
YYYY_作者_中文主题.md
```

示例：

```text
2001_Fleming_波动率择时经济价值.md
```

模块想法：

```text
IYYYYMMDD-NNN_中文主题.md
```

概念词条：

```text
中文概念名.md
```

英文缩写写入 aliases，不优先作为文件名。已经广泛使用的英文缩写例外，例如 B3 gate、A2-slope004。

策略词条：

```text
中文策略名.md
```

示例：

```text
ETF双池动量轮动.md
小市值日内做T.md
```

### 4.8 中文优先规则

必须使用中文的部分：

1. 文件中文短标题。
2. 方向页正文。
3. 实验记录正文。
4. 决策摘要。
5. 结论、风险备注、下一步。
6. Agent 操作规范和接手指南。
7. 资产清单 Markdown 摘要。

可以使用英文或 ASCII 的部分：

1. 稳定编号。
2. 脚本参数名。
3. 配置字段名。
4. 平台 run_id。
5. manifest 字段。
6. 已形成行业惯例的缩写。

中英混用规则：

```text
第一次出现：B3 gate（B3 确认门）
后续出现：B3 gate
```

不要出现只有英文缩写、没有中文解释的标题。

### 4.9 分类分区允许重新调整

二期重构允许重新调整目录分区，但必须遵守先登记、再移动、后校验。

允许调整的情况：

1. 文档类型和所在目录不一致。
2. 当前文档同时承担入口、总结、实验和决策多种职责。
3. 归档文件仍标记为启用，影响当前事实源。
4. 信念文件混在方向目录，影响驾驶舱自动聚合。
5. 专题总结应当成为方向页，但仍在归档或旧目录描述中。
6. 模板或规范仍引用旧目录，导致新 Agent 接手路径错误。

调整方式：

```text
第一步：在资产清单和迁移映射中登记原路径、目标路径、调整原因。
第二步：先修改 frontmatter 的 type、status、archive_status。
第三步：更新引用该文档的入口、方向页和决策卡。
第四步：再移动文件。
第五步：运行健康检查和抽样追溯检查。
```

迁移映射必须至少记录：

```text
old_path
new_path
file_type_before
file_type_after
move_reason
decision_or_governance_record
moved_at
verified_at
```

### 4.10 错放、重复和错误内容治理规则

错放文档处理：

1. 不直接删除。
2. 先判断它是活文档、记录文档、模板、归档材料还是错误副本。
3. 活文档移动到当前对应目录。
4. 历史材料进入归档并标记 archive。
5. 旧入口必须写明“不作为当前入口”。

重复文档处理：

1. 内容完全重复：保留一份 canonical 文档，另一份进入归档或删除前先登记。
2. 内容部分重复但时间不同：保留较新版本为 canonical，旧版本标记 superseded。
3. 名称重复但职责不同：必须改名，避免 Obsidian 和脚本按文件名解析混淆。
4. 概念页和方向页同名时，优先保留方向页中文名，概念页改成“概念：xxx”或更具体的术语名。

错误内容处理：

1. 明确错误但有历史意义：不覆盖删除，在原文附近追加“已废弃原因”或迁入归档。
2. 明确错误且会误导当前研究：在当前入口和方向页中取消引用，改指 canonical 文档。
3. 模板中的旧路径、旧字段、旧状态枚举必须直接修正，不保留错误示例。
4. 健康检查中的历史段落可以保留，但必须增加“历史口径”说明。

canonical 文档要求：

```text
每个方向只有一个当前方向页
每个实验编号只有一个 canonical 实验记录
每个决策编号只有一张 canonical 决策卡
每个概念只有一个 canonical 概念词条
每个新手入口只有一个 canonical 入口
```

2. topic_id 表示模块或研究节点，例如 R010-B5S4。
3. direction_id 表示所属研究方向，例如 R010-B5。
4. 路径字段使用列表，不使用分号字符串，不使用通配符。
5. 历史通配符可以保留在正文，但新增资产清单必须展开为实际路径。

### 5.3 result_status 枚举

不是每一次实验都必须有回测结果，但每一次实验都必须有结果状态。

建议枚举：

```text
preregistered_no_run
backtest_result
readonly_audit_result
shadow_observation_result
config_dry_run
failed_or_aborted
parked_waiting_sample
governance_only_no_market_result
```

含义：

1. preregistered_no_run：只完成预注册，尚未执行。
2. backtest_result：有回测结果。
3. readonly_audit_result：只读审计，不回测。
4. shadow_observation_result：影子观察结果。
5. config_dry_run：配置或可运行性检查。
6. failed_or_aborted：执行失败或中止，也要记录失败日志和原因。
7. parked_waiting_sample：等待新样本，暂不继续执行。
8. governance_only_no_market_result：治理文档或规范类记录，不应伪装成市场实验。

关键原则：

1. 每一次回测不一定是一篇实验记录。
2. 每一个正式实验必须能解释结果在哪里，或为什么没有结果。
3. 每一个平台 run 必须能被某个实验记录、决策卡或资产清单引用。
4. 如果一篇记录长期没有结果，也要有明确的 result_status 和 next_step。

## 六、资产保存结构

### 6.1 研究库保存什么
