~~~markdown
## 一、核心设计原则

### 1. 四个层，严格分离

```
第0层 — 导航层（少量入口文件）
  驾驶舱 + 变更日志。方向状态以 Dataview 聚合为主，最高优先级等人工队列允许手动维护。
  回答："现在什么状态？我不在的时候发生了什么？"

第1层 — 方向层（每个研究方向一个文件）
  方向的完整故事。从诞生到当前状态的推理链。
  回答："这个方向研究了什么？为什么是这个结论？"

第2层 — 证据层（保留证据，谨慎修订）
  实验记录 + 决策卡片。底层原始证据。
  回答："第27号实验具体数据？那天做了什么决定？"

第3层 — 参考层（稳定，低频更新）
  概念词条、论文笔记、方法库、策略定义。
  回答："B3 gate 是什么？这篇论文说了什么？"
```

### 2. 两种文档，职责分明

| | 活文档（Living） | 记录文档（Record） |
|---|---|---|
| **更新方式** | 状态变化时**替换**当前摘要 | 事件发生时**追加**；治理补链、格式修复可修订，但不得改写历史证据和结论 |
| **内容** | 当前状态 + 推理链 | 一次实验的完整数据 + 一次决策的完整判据 |
| **例子** | 方向页面、驾驶舱 | 实验记录、决策卡片、变更日志 |
| **谁写** | 方向收束时重写 | 每次实验/决策后立刻写 |

### 3. 状态驱动

- 每个方向的当前状态写在**方向页面 frontmatter** 里
- 驾驶舱用 **Dataview 自动读取**，不手动维护表格
- 状态变了 → 改方向页 frontmatter → 驾驶舱自动更新对应 Dataview 区块 → 必要时追加变更日志并更新最高优先级人工队列

## 三、每种文档的 Frontmatter Schema

### 方向页面（`🧭 方向/`）

```yaml
---
type: 方向
status: live | in_progress | revise | park | killed | waiting
started: 2026-05-30    # 方向首次启动日期
updated_at: 2026-06-03    # 最后一次状态变更日期
confidence: 中          # 当前研究信念置信度
strategy: "[[ETF双池动量轮动]]"  # 关联策略
summary: "5轮淘汰后形成三层分级防守系统"  # 一句话总结
blocker: ""             # 阻塞项（仅 status=park/revise 时填写）
last_decision: "[[D20260602-R010B4-tiered-v2实盘启用]]"
next_step: "核验实盘首个交易日日志"
---
```

### 实验记录（`🧪 实验/`）

```yaml
---
type: 实验
direction: "[[ETF双池防守系统]]"    # 所属方向
stage: preregistration | smoke | four_segment | counterfactual | attribution
verdict: promote | revise | kill | park | observe  # 本轮实验结果
started: 2026-06-02
completed: 2026-06-02
baseline_return: "+1468.34%"
test_return: "+1534.91%"
baseline_mdd: "-27.13%"
test_mdd: "-25.33%"
data_path: "results/v2/research/R010-B4/"   # 数据路径，不在 Markdown 里放大表
---
```

```yaml

## 四、驾驶舱设计（Dataview 自动聚合）

```markdown
# 🟢 实盘运行
dataview: FROM "🧭 方向" WHERE status = "live"

# 🟡 进行中
dataview: FROM "🧭 方向" WHERE status = "in_progress" OR status = "revise"

# 📌 暂停 / 等待
dataview: FROM "🧭 方向" WHERE status = "park" OR status = "waiting"

# ❌ 已证伪
dataview: FROM "🧭 方向" WHERE status = "killed"

# 📖 概念状态
dataview: FROM "📖 概念" WHERE status = "实盘" OR status = "观察中"
```

每个 Dataview 区块自动从 frontmatter 读取 `summary`、`blocker`、`next_step` 等字段，状态变了只需要改方向页面的 `status` 字段。

## 五、变更日志设计

`📋 变更日志.md` — 每次方向状态变化时，**在顶部追加一行**：

```markdown
## 2026-06-03

| 时间 | 方向 | 旧状态 | 新状态 | 原因 |
|------|------|--------|--------|------|
| 08:30 | 非线性仓位控制 | in_progress | park | R5反事实未通过，等独立样本 |
| 08:15 | ETF双池防守系统 | in_progress | live | 用户授权后实盘启用 |
```

方向页面里的"完整故事"解释**为什么**；变更日志只记录**什么变了**。两者互补。

## 六、研究生命周期

```
灵感 ──→ 信念 ──→ 预注册 ──→ 实验 ──→ 决策 ──→ 方向收束
  │        │        │         │        │         │
  │        │        │         │        │         └→ 重写方向页面（活文档）
  │        │        │         │        └→ 新建决策卡片（记录文档）
  │        │        │         └→ 新建实验记录（记录文档）
  │        │        └→ 实验记录：stage = preregistration
  │        └→ 💡 想法/ 或 📚 文献/ （参考层）
  └→ 💡 想法/ （参考层）
```

### 每个阶段的产物

| 阶段 | 必须产物 | 存放位置 | 文档类型 |
|------|---------|---------|---------|
| 灵感 | 文献笔记 或 模块想法 | `📚 文献/` 或 `💡 想法/` | 记录 |
| 形成信念 | 方向页面（初始版） | `🧭 方向/` | 活文档 |
| 实验前 | 实验记录（预注册） | `🧪 实验/` | 记录 |
| 实验后 | 实验记录（更新结论） | `🧪 实验/` | 记录 |
| 方向决策 | 决策卡片 | `🔀 决策/` | 记录 |
| | 方向页面（更新状态） | `🧭 方向/` | 活文档 |
| | 变更日志（追加一行） | `📋 变更日志.md` | 记录 |
| 方向收束 | 方向页面（重写完整故事） | `🧭 方向/` | 活文档 |

## 九、执行风险

| 风险 | 缓解 |
~~~
