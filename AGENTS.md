# Agent 操作规范

本文件是所有接手本研究库的 Agent 必读入口。无论历史对话中说过什么，研究库内文档才是当前事实源。

## 语言要求

- 所有交流、计划、总结、文档、提交说明和代码注释默认使用中文。
- 遇到英文论文、代码变量或平台日志，可以保留原文关键名词，但解释必须用中文。
- 面向新手的实验总结禁止只写指标结论，必须解释研究方向和决策原因。

## 编码要求

- Agent 读写本库文本文件时始终优先使用 UTF-8。
- Markdown、CSV、JSON、Canvas、YAML、PowerShell 脚本和 Git 配置文本都必须能被 UTF-8 正常解码。
- 除非兼容 Windows PowerShell 5.1 等工具确有必要，新增文本文件优先使用 UTF-8 无 BOM。
- 禁止使用 ANSI、GBK、UTF-16 等会导致 Obsidian、GitHub 或跨系统 Agent 解析不一致的编码。
- 提交前执行 `tools/Test-ResearchRepo.ps1`，确认核心文本文件通过 UTF-8 检查。

## 系统边界

| 系统 | 职责 |
| --- | --- |
| 本研究库 | 方向、假设、文献、灵感、实验记录、决策、台账、术语、交接 |
| 回测平台 | 策略代码、回测配置、数据、执行、结果、大型报告、质量门禁 |
| 旧研究库 | 只作为待迁移资料源，不作为新架构依据 |

禁止把大型净值、交易明细、参数扫描结果、原始数据和论文 PDF 直接提交到研究库。

## 每轮实验硬规则

正式实验前必须写清楚：

- 研究方向 ID。
- 本次假设。
- 实验前预测。
- 基准对照。
- 竞争性解释。
- 证伪条件。
- 计划使用的平台配置路径和结果路径。

正式实验后必须补齐：

- 实际观察。
- 和预测一致或不一致的地方。
- 支持证据。
- 反对证据。
- 对新手的短总结。
- 下一步。
- 是否需要研究决策卡。

没有实验前预测的回测结果，只能记为“探索观察”，不得直接升级为正式结论。

## 未来函数与过拟合硬规则

- 未完成未来函数审计的实验，不得 `promote`。
- 未完成样本外、成本扰动、参数敏感性或消融检查的策略改进，不得写成“已验证有效”。
- 子代理可以检查字段和路径，但不得独立判断“无未来函数”。
- 看结果后新增参数网格，必须作为新实验重新预注册。
- Shadow、probe、observe 结果不得直接升级为默认交易逻辑。
- 如果负控或错位信号仍表现良好，优先怀疑泄漏、未来函数或过拟合，而不是解释为策略强。

## 方向、模块和实验关系

- 策略级研究方向使用 `RD`，例如“双池轮动策略”。
- 策略下的进攻、核心、防御、执行、诊断等模块也可以是 `RD`，但必须填写 `parent_rd_id` 指向策略级方向。
- 实验记录使用 `EX`，必须至少关联一个 `rd_id`。
- 路线判断使用 `DEC`，必须关联受影响的 `rd_id` 和关键 `ex_id`。

这样做的目的不是增加文档数量，而是让后来者能快速看懂“这个模块为什么被保留、修改、暂停或淘汰”。

## 命名与新建文件

禁止使用人工递增顺序号作为唯一编号。新文件优先通过工具生成：

```powershell
powershell -ExecutionPolicy Bypass -File tools/New-ResearchItem.ps1 -Type Experiment -Title "实验标题" -ParentId "RD-..."
```

如必须手工创建，文件名格式必须是：

```text
<ID>_<中文短标题>.md
```

ID 格式见 `08_方法论/命名与编号规范.md`。

## 平台命令边界

Agent/CLI 回测必须在 WSL 内运行。常用形式：

```bash
wsl -- bash -c "cd /mnt/e/量化平台_V1.4.0 && PYTHONPATH=src python3 src/run_v2_backtest.py --config configs/research/<rd_id>/<ex_id>/<config>.json"
```

不要自动执行数据更新。只有用户明确要求更新数据时，才按平台文档运行外部数据更新入口。

## 子代理边界

可以用子代理降低高等级模型用量，但不能降低研究质量。

- `gpt-5.3-codex-spark`：只做检索、清单、路径核对、格式检查、字段摘录。
- `gpt-5.4`：做证据归纳、草稿整理、实验记录初稿。
- 主线程或 `gpt-5.5`：负责研究假设、证伪规则、未来函数审计、实盘边界和最终决策。

子代理不得独立决定 `promote`、`revise`、`park`、`kill`，也不得独立修改实盘默认开关。

默认规则：涉及实验复核、路径核对、证据归纳、模板检查、台账审计、Git 发布或平台资产盘点时，主控 Agent 应优先拆出至少一个可并行子代理任务。若本轮不调用子代理，必须在相关实验、决策或工作总结中写明“子代理豁免原因”。

任务开始前，主控应先写出一句子代理计划：

```text
子代理计划：调用/不调用；任务代号；模型；交付物；豁免原因。
```

任务完成后，凡影响实验或决策的子代理调用，都必须写入文档的“子代理调用记录”区块，并同步 `01_台账/子代理调用台账.csv`。缺少调用记录且缺少豁免原因的 `completed` 实验或决策，视为流程不完整。

## 完成工作前检查

结束前至少检查：

- 新增文件是否有唯一 ID。
- `01_台账/` 是否同步更新。
- `00_入口/研究驾驶舱.md` 是否需要更新。
- `00_入口/研究路线图.canvas` 是否需要更新。
- 新术语是否进入 `09_术语库/术语库.md`。
- 核心资产是否补齐 Obsidian 双向链接。
- 子代理调用是否已记录到正文和 `01_台账/子代理调用台账.csv`，或是否写明豁免原因。
- 平台配置和结果路径是否只登记路径，不复制大文件。
- 若改变研究路线，是否新增或更新 `05_研究决策/`。

## Obsidian 边界

人类接手时默认使用 Obsidian 阅读本库。Agent 可以维护 Obsidian 配置，但不得提交 workspace、缓存、token 或插件 `data.json`。涉及 Obsidian 的改动先读 `08_方法论/Obsidian使用规范.md`。

## 双向链接边界

Obsidian 双向链接应充分使用，但不得替代 ID、frontmatter 和台账。核心资产需要“关联链接”区块，方向、实验、决策、文献和灵感之间要能通过 Backlinks 面板互相追溯。具体规则见 `08_方法论/Obsidian双向链接规范.md`。

## 路线图边界

全局路线图使用 Obsidian Canvas：`00_入口/研究路线图.canvas`。Mermaid 只用于 10 个以内节点的小型缩略图。节点变多后，拆分子 Canvas，不要把所有实验塞进一张 Markdown 图。
